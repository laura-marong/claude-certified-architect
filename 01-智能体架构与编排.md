# Claude 认证架构师考试 — 领域 1：智能体架构与编排（权重：约 25%）

## 概述

本领域涵盖使用 Claude Agent SDK 设计和构建智能体系统。主题包括智能体循环、多智能体编排、钩子（Hooks）、工作流、会话管理，以及面向生产级 AI 应用的任务分解。

---

## d1.1 — 智能体循环与核心 API

### 核心概念

- **智能体循环生命周期**：`stop_reason` 字段决定循环是继续还是结束——两个关键值是 `'tool_use'` 和 `'end_turn'`
- **工具结果追加**：工具执行后，其结果会被追加回对话中，供下一轮迭代使用
- **SDK 控制流**：Agent SDK 自动管理循环，但理解底层机制对考试至关重要
- 只要 `stop_reason` 等于 `'tool_use'`，循环就持续；当它变为 `'end_turn'` 时，循环终止

### 循环工作原理（5 步流程）

1. 向 Claude 发送一条消息，附带已定义的工具
2. Claude 响应：要么返回文本（完成），要么返回工具调用（需要执行操作）
3. 如果调用了工具，执行它并将结果附加到对话中
4. 将更新后的对话重新发送给 Claude
5. 重复上述过程，直到 Claude 仅回复文本（不再有工具调用）

API 响应中的 `stop_reason` 字段被描述为控制循环的"唯一可靠信号"。

### 反模式

- 通过解析自然语言输出来决定是否继续，而不是检查 `stop_reason`
- 使用任意迭代次数上限作为主要停止机制
- 通过检查助手文本内容来判断循环是否终止

### 代码示例概要

提供的 `agentic-loop.py` 示例展示了：

- 一个 `while True` 无限循环，通过 `client.messages.create()` 创建消息，传入 model、max_tokens、tools 和 messages 参数
- 关键分支逻辑：如果 `stop_reason == "end_turn"`，退出循环；如果 `stop_reason == "tool_use"`，提取工具调用块，执行它，然后将助手响应和一个 `tool_result` 用户消息一起追加回消息列表

### 考试提示

考试会提供 3–4 个循环终止选项。正确的答案始终涉及检查 `stop_reason`。干扰项包括解析文本内容、设置迭代次数限制，或监控 token 数量。

## d1.2 — 多智能体编排

### 核心模式

- **轮辐式架构（Hub-and-Spoke）**：一个中心协调器将任务委派给专门的子智能体
- **上下文隔离**：每个子智能体维护自己的上下文，不与其他智能体直接共享状态
- **用于生成子智能体的 Task 工具**：协调器的 `allowedTools` 必须包含 `'Task'` 才能创建子智能体
- **并行执行**：在单次响应中发出多个 Task 调用，可使子智能体同时工作
- **fork_session**：创建分支会话以进行并行探索，而不污染主上下文

### 轮辐式架构为何有效

- 上下文隔离确保每个子智能体只接收与其任务相关的内容
- 聚焦的工具访问意味着每个子智能体拥有大约 4–5 个与其专业领域相关的工具
- 并行执行允许多个子智能体同时运行
- 清晰的综合：协调器合并结果而不携带探索过程中的噪声

### Task 工具

Task 工具用于生成子智能体。每次 Task 调用指定子智能体的提示词、工具和上下文。单次响应中发出的多个调用将并行运行。

### 上下文传递规则

仅传递与每个子智能体任务相关的上下文。切勿共享完整的协调器对话历史——这样做会浪费 token 并让子智能体被无关细节干扰。

### 代码示例概要

`hub-and-spoke.py` 示例创建了：

- 一个**协调器**智能体，拥有 `Task` 工具以及综合和格式化工具
- 一个**市场研究员**子智能体，限定为四个工具：web_search、read_doc、extract_data、format_citation
- 一个**技术分析师**子智能体，限定为另外四个工具：read_code、grep_patterns、analyze_deps、format_report
- 协调器以一条明确指示每次子任务进行显式委派并传递上下文的提示词运行

### 反模式

- 任务分解过于狭窄，导致子智能体之间存在覆盖空白
- 与每个子智能体共享完整的协调器上下文（上下文污染）
- 委派时未能提供明确的上下文

### 考试提示

上下文隔离是重点考查内容。如果某个答案选项将完整的协调器上下文共享给子智能体，则该选项是错误的。每个子智能体应仅接收与其自身子任务相关的上下文。

## d1.3 — 钩子与编程式强制执行

### 核心概念

- **PostToolUse 钩子**：拦截并修改工具输出，适用于数据规范化
- **编程式强制执行**：用于关键业务规则——确定性的，而非概率性的
- **基于提示词的引导**：仅适用于软性偏好和风格建议
- **基于钩子的阻断**：示例为阻断超过 $500 的退款并将其重定向到升级处理

### 关键区别

- **钩子（Hooks）**= 确定性的（100% 可靠）→ 用于关键业务规则、合规和安全
- **提示词（Prompts）**= 概率性的（模型可能忽略）→ 用于风格偏好和软性指南

### 钩子类型

- **PreToolUse**：在工具执行前拦截；可以阻断、修改参数或添加验证
- **PostToolUse**：在执行后拦截；可以修改输出、规范化数据或触发副作用

给出的示例：在提示词中设置 $500 退款上限可能被忽略，但 PostToolUse 钩子可以保证阻断生效。

### 有效的升级触发条件

- 客户明确请求人工介入
- 检测到策略空白（没有规则覆盖当前情况）
- 任务超出智能体能力范围
- 超出业务阈值（例如，退款超过 $500）

### 无效的升级触发条件（反模式）

- 负面情绪（情绪不等于任务复杂度）
- 自报低置信度（模型置信度不可靠）
- 基于情绪的升级（普遍不推荐）

### 代码示例概要

`hooks.py` 示例定义了一个 `refund_limit_hook` 函数：

- 检查工具名称是否为 `"process_refund"`
- 从工具输入中获取金额
- 如果金额超过 500，返回包含原因、升级操作和上下文信息（customer_id、requested_amount、agent_limit）的阻断响应
- 否则原样返回工具输出
- 智能体以 `hooks={"PostToolUse": [refund_limit_hook]}` 的方式实例化

### 考试提示

当考试询问关于强制执行关键业务规则（退款上限、数据访问、合规）时，正确答案始终是编程式钩子（programmatic hooks），绝不是提示词指令。

## d1.4 — 会话管理与工作流

### 核心概念

- **--resume 标志**：以完整保留的上下文继续之前的会话
- **fork_session**：分支探索而不污染主上下文
- **命名会话**：使用 `--session-name` 实现有组织的多会话工作流
- **过期上下文检测与缓解**：长时间运行会话中的关键问题
- **提示链 vs. 动态自适应分解**：选择取决于任务的可预测性

### 关键会话操作

- **恢复（Resume）**：通过 `--resume` 以完整上下文继续之前的会话
- **分支（Fork）**：使用 `fork_session` 创建分支进行探索——分支中的更改不影响主会话
- **命名会话（Named sessions）**：使用 `--session-name` 启动新的命名会话，用于有组织的工作流

### 过期上下文

长时间运行会话中的一个关键风险是早期获取的数据变得过时。缓解策略包括定期重新获取关键数据和使用暂存文件（scratchpad files）。

### 任务分解策略

- **提示链（Prompt chaining）**：最适合可预测的线性任务，步骤顺序固定，每步的输入输出定义清晰
- **动态自适应分解（Dynamic adaptive decomposition）**：最适合不可预测的复杂任务，智能体根据中间结果决定下一步；当任务复杂度未知或结果可能改变处理方式时优先选用

### 代码示例概要

`session-management.sh` 示例演示了：

- `claude --resume` — 恢复最近的会话
- `claude --resume --session-name "feature-auth-redesign"` — 恢复指定的命名会话
- `claude fork_session --reason "探索替代 API"` — 分支探索（继承上下文但分叉）
- `claude --session-name "sprint-47-backend"` — 启动新的命名会话

### 反模式

- 在长时间会话中忽略过期上下文
- 对需要动态适应的任务使用静态提示链

### 考试提示

如果任务不可预测或含有条件分支，动态自适应分解是正确的选择。如果是固定的线性管道，提示链适用。要清楚何时使用 `fork_session`，何时使用 `--resume`。

## 考试提示汇总

1. 始终检查 `stop_reason` 来控制循环——绝不要解析自然语言
2. 对业务规则使用编程式钩子，对偏好使用提示词
3. 子智能体需要明确的上下文——不要假设它们继承了协调器的知识
4. 理解 `fork_session` 与 `--resume` 的区别及其各自适用场景
