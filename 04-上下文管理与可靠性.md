# Claude 认证架构师考试 — 领域 5：上下文管理与可靠性（权重：约 15%）

## 概述

在生产系统中有效管理上下文。涵盖渐进式摘要风险、上下文定位、升级模式、错误传播、上下文退化、人工审核及信息溯源。

---

## d5.1 上下文优化与定位

### 概要

通过战略性定位、裁剪和摘要技术优化上下文窗口的利用，同时避免常见陷阱。

### 上下文管理技术

- **渐进式摘要风险**：重复摘要可能导致重要细节丢失
- **"中间丢失"效应**：长上下文中间部分的信息被模型回忆起的可能性更低
- **"案件事实"块**：结构化参考区段，用于保存关键信息
- **裁剪冗长工具输出**：去除噪声，同时保留必要数据
- **位置感知排序**：将最重要的信息放在上下文开头和结尾

### 应避免的反模式

- 对关键细节进行渐进式摘要而不保留原始信息
- 在长上下文窗口中忽视"中间丢失"效应

### 深度解析

上下文管理的核心是在有限的上下文窗口内最大化利用空间，同时保护关键信息。两个关键概念主导此领域：

**1. 渐进式摘要风险**

渐进式摘要压缩对话历史以节省空间。看似高效，实则悄然破坏关键细节。例如：原始信息包含客户姓名、账号、订单号、具体金额和促销代码——经过两次摘要后，这些信息全部丢失。

**2. "中间丢失"效应**

研究表明，模型对长上下文中间部分信息的回忆概率更低，而对开头和结尾的信息关注更多。

**解决方案："案件事实"块**

不进行摘要，而是将关键信息保存在一个不可变的结构化块中，放置在上下文开头（高回忆概率位置）。此块永不摘要或压缩，包含所有必要的参考数据。

### 代码示例 — 不可变参考块

```markdown
## 案件事实（不要摘要——直接引用）

| 字段           | 值                                        |
|----------------|-------------------------------------------|
| 客户           | 张三                                      |
| 账户 ID        | ACC-12345                                 |
| 订单           | #98765                                    |
| 预期价格       | ¥99.99（促销码 SUMMER2026）               |
| 实际扣款       | ¥150.00                                   |
| 超额扣款       | ¥50.01                                    |
| 客户起始年份   | 2019（7 年老客户）                        |
| 优先级         | 高（长期客户 + 超额扣款）                 |

## 规则
- 始终以"张先生"称呼客户
- 本案符合即时解决条件
- 退款金额（¥50.01）在 Agent 退款上限（¥500）以内
```

### 对比：反模式 vs 正确做法

| 反模式 ❌ | 正确做法 ✅ |
|-----------|------------|
| 渐进式摘要丢失关键细节（到第 10 轮时，姓名、账号、订单、金额全部丢失） | 案件事实块始终可用，永不摘要，保持完整 |

### 考试提示

如果考试问如何在长对话中保留关键客户细节，答案永远是"案件事实"块，而非渐进式摘要。

## d5.2 升级与错误传播

### 概要

设计升级模式和错误传播策略，为恢复或人工干预提供足够的上下文。

### 升级与错误处理要点

- **升级触发条件**：客户明确要求、策略空白——而不仅仅是情绪信号
- **结构化错误上下文 vs 通用错误**：始终包含尝试过的操作内容
- **访问失败 vs 空结果**：区分"无法检查"与"已检查但未找到任何内容"
- **协调器升级前先尝试本地恢复**：尽量先就地修复
- **部分结果 + 尝试过的操作**：即使失败也要始终报告进度

### 应避免的反模式

- 基于情绪的升级（情绪不等于任务复杂度）
- 丢失原始错误上下文的通用错误传播
- 静默吞掉错误而不携带上下文进行升级

### 深度解析

升级和错误传播模式决定了故障如何在系统中流动。处理不当意味着要么用琐碎问题淹没人工，要么静默丢弃关键故障。

**有效的升级触发条件：**

- 客户明确要求转接人工
- 检测到策略空白（现有规则未涵盖的情况）
- 任务超出 Agent 能力（需要 Agent 没有的访问权限）
- 超出业务阈值（例如退款超过 ¥500，由钩子处理）
- 经过合理恢复尝试后反复失败

**无效的升级触发条件（考试反模式）：**

- **负面情绪**——一个仅需改地址的愤怒客户不应升级。情绪不等于任务复杂度
- **自报置信度**——模型自身的置信度评估不可靠

**多 Agent 系统中的错误传播：**

当子 Agent 失败时，必须向协调器报告结构化上下文：尝试了什么、发生了什么错误（带类别和可重试性）、是"访问失败"还是"空结果"。协调器随后决定：重试、使用替代方案或升级。

### 代码示例 — 结构化升级逻辑

```python
def should_escalate(context):
    """判断是否需要人工介入。"""

    # 有效的升级触发条件
    if context.customer_requested_human:
        return True, "客户明确请求转接人工"

    if context.policy_gap_detected:
        return True, "无现有策略覆盖当前情况"

    if context.amount > AGENT_REFUND_LIMIT:
        return True, f"金额 {context.amount} 超出 Agent 上限"

    if context.retry_count >= MAX_RETRIES:
        return True, "已达最大重试次数"

    # 无效的触发条件——绝不要使用以下方式
    # if context.sentiment == "negative":  # 错误！
    #     return True  # 情绪 ≠ 复杂度

    # if context.model_confidence < 0.7:  # 错误！
    #     return True  # 自报置信度不可靠

    return False, None
```

### 对比：反模式 vs 正确做法

| 反模式 ❌ | 正确做法 ✅ |
|-----------|------------|
| 基于情绪升级（愤怒客户仅需改地址并不需要人工） | 基于客观标准升级（客户明确请求、策略空白、金额超限） |

### 考试提示

基于情绪和置信度的升级在考试中始终是错误的。有效触发条件为：客户明确请求、策略空白、能力限制和业务阈值。

## d5.3 上下文退化与长时会话

### 概要

处理长运行会话中的上下文退化。使用暂存文件、`/compact` 以及子 Agent 委托来维持质量。

### 管理长时会话的方法

- **上下文退化**：长时会话中，随着上下文空间填满，响应质量下降
- **暂存文件**：将重要状态持久化到外部文件，以便在上下文重置后恢复
- **/compact**：压缩对话历史以回收上下文空间
- **子 Agent 委托**：将冗长探索委托给子 Agent，保持协调器上下文清洁
- **崩溃恢复清单**：持久化状态文件，支持会话恢复

### 应避免的反模式

- 运行长时会话却不监控上下文退化
- 对重要中间状态不使用暂存文件

### 深度解析

长运行 Agent 会话会遭受**上下文退化**——随着对话增长和上下文窗口填满，响应质量下降。

**上下文退化的症状：**

- Agent 忘记之前的指令或约束
- 响应变得不够聚焦、更泛化
- 工具选择准确度下降
- Agent 可能重复已完成的工作

**缓解策略（考试重点）：**

1. **/compact** — 压缩对话历史以回收上下文空间。当上下文变长但任务尚未完成时使用
2. **暂存文件** — 将关键中间状态持久化到外部文件。这些文件在上下文压缩和会话边界上都能留存
3. **子 Agent 委托** — 将冗长探索任务委托给子 Agent。子 Agent 的上下文吸收探索噪声，仅将综合结果返回协调器
4. **位置感知上下文排序** — 将最重要信息放在上下文开头和结尾（高回忆概率位置），次要信息放中间
5. **分层指标（按文档类型追踪）** — 聚合准确率可能掩盖每类文档的失败。需要按文档类型追踪准确率
6. **信息溯源** — 始终保留信息的来源和置信度级别。追踪每条数据的来源及其可靠性

### 代码示例 — 退化缓解

```python
# 策略 1：暂存文件保存持久化状态
agent.run("""
在开始复杂分析之前：
1. 创建暂存文件：progress.md
2. 发现关键发现时即时记录
3. 每个主要步骤完成后更新 progress.md
4. 如果上下文变长，使用 /compact
5. /compact 后，重新读取 progress.md 恢复上下文
""")

# 策略 2：冗长任务委托给子 Agent
coordinator = Agent(tools=[Task, read_scratchpad, summarize])
coordinator.run("""
对于此代码库分析：
1. 将逐文件分析委托给子 Agent
   （使冗长探索不进入协调器上下文）
2. 子 Agent 将发现写入暂存文件
3. 协调器读取综合后的发现
4. 协调器生成最终报告
""")

# 策略 3：分层指标
def track_accuracy(results):
    """按文档类型追踪，而非仅看聚合数据。"""
    by_type = {}
    for r in results:
        doc_type = r["document_type"]
        if doc_type not in by_type:
            by_type[doc_type] = {"correct": 0, "total": 0}
        by_type[doc_type]["total"] += 1
        if r["is_correct"]:
            by_type[doc_type]["correct"] += 1

    for doc_type, stats in by_type.items():
        accuracy = stats["correct"] / stats["total"] * 100
        print(f"{doc_type}: {accuracy:.1f}%")
```

### 对比：反模式 vs 正确做法

| 反模式 ❌ | 正确做法 ✅ |
|-----------|------------|
| 仅使用聚合指标——总准确率 95% 看起来很优秀，但实际上发票准确率仅 70%，问题被掩盖 | 按文档类型的分层指标——揭示各类别隐藏的失败 |

### 考试提示

聚合指标掩盖每类文档的失败是考试重点概念。正确答案始终是按文档类型追踪准确率，而非仅看整体准确率。

## d5.4 人工审核与信息溯源

### 概要

设计人机协作审核系统，并通过声明-来源映射和时间数据保持信息溯源。

### 人工审核与溯源要点

- **分层抽样**：跨不同类别审核样本，而非仅随机选择
- **字段级置信度**：为单个数据字段提供置信度指示
- **按文档类型的准确率**：按文档类别追踪表现，而非仅看聚合数据
- **声明-来源映射**：将每个输出声明链接到其来源以保证可追溯性
- **时间数据**：保留时间戳和版本信息以保证时效性
- **冲突标注**：明确标记冲突来源，而非静默选择其一

### 应避免的反模式

- 仅使用聚合准确率指标，掩盖每类文档的失败
- 不维护声明-来源映射以保证可追溯性
- 静默解决来源冲突而非进行标注

### 深度解析

**信息溯源**意味着追踪每条数据的来源及其可靠性。这对多 Agent 系统至关重要，因为不同子 Agent 从不同来源贡献信息。

**为什么溯源很重要：**

- 当两个子 Agent 提供冲突信息时，协调器需要知道哪个来源更可靠
- 审计追踪需要知道哪些数据来自哪个来源
- 下游决策依赖数据质量——来自已验证数据库的数字比从 PDF 提取的更可靠

**需要追踪的溯源元数据：**

- **来源**：数据来自哪里？（API、数据库、文档、网页）
- **置信度**：来源有多可靠？（已验证、已提取、已推断、已估计）
- **时间戳**：数据何时被获取？
- **Agent**：哪个子 Agent 提供了该数据？

**人机协作检查点：**

对于关键决策，系统应暂停并向人工呈现：

- 待做出的决策
- 支持每个选项的数据（含溯源）
- 建议的操作及原因
- 批准/修改/拒绝的方式

### 代码示例 — 信息溯源追踪

```python
from dataclasses import dataclass
from datetime import datetime
from typing import Literal

@dataclass
class DataWithProvenance:
    value: str | float | dict
    source: str                    # "customer-db"、"invoice-pdf"、"web"
    confidence: Literal["verified", "extracted", "inferred", "estimated"]
    retrieved_at: datetime
    agent_id: str                  # 提供该数据的子 Agent

def resolve_conflict(data_points: list[DataWithProvenance]):
    """当子 Agent 数据冲突时，使用溯源信息决策。"""

    confidence_rank = {
        "verified": 4,
        "extracted": 3,
        "inferred": 2,
        "estimated": 1,
    }

    best = max(data_points, key=lambda d: confidence_rank[d.confidence])

    log_conflict(
        chosen=best,
        alternatives=data_points,
        reason=f"选择 {best.source}（置信度：{best.confidence}）"
    )

    return best
```

### 对比：反模式 vs 正确做法

| 反模式 ❌ | 正确做法 ✅ |
|-----------|------------|
| 无溯源追踪——两个子 Agent 给出冲突的收入数据，无法判断信任哪一个，随意选择 | 附带溯源追踪——明确来源和置信度，基于已验证数据库优先于提取的 PDF |

### 考试提示

当子 Agent 提供冲突数据时，正确答案始终涉及追踪信息溯源并用其解决冲突。没有溯源的随意选择始终是错误的。

## 领域 5 考试提示汇总

1. 渐进式摘要丢失关键细节——应使用"案件事实"块替代
2. 情绪 ≠ 升级决策的复杂度
3. 始终区分访问失败与真正的空结果
4. 按文档类型追踪准确率，而非仅看聚合数据
