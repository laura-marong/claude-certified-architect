## d4.1 显式标准与指令设计

### 概要

编写具有**显式、可衡量标准**的提示，而非模糊指令。理解误报如何影响开发者信任。

### 提示设计原则

- **显式标准优于模糊指令**：例如用"标记超过 50 行的函数"替代"标记过长的函数"
- **误报影响**：过多的误报会侵蚀开发者对系统的信任
- **具体性**可减少歧义并提高跨运行的一致性
- **可衡量的标准**使得输出质量的自动化验证成为可能

### 应避免的反模式

- 模糊指令，如"让它更好"或"改进代码"
- 未考虑误报对下游的影响

### 深度解析

生产级提示需要**显式、可衡量的标准**，而非模糊指令。这是贯穿多个考试场景的基本原则。

**模糊性在生产中失败的原因：**

- "让它更好"——更好指什么？更快？更简洁？更短？
- "找出问题"——什么算问题？每个风格细节？仅限 bug？
- "要彻底"——导致过度标记、误报、信任侵蚀

**误报问题：**

当代码审查工具标记过多非真正问题时，开发者会开始忽略**所有**标记——包括真正的问题。这称为**告警疲劳**，在考试中会直接考查。

**解决方案——可衡量的标准：**

用"标记超过 50 行代码的函数"替代"标记过长的函数"。用"识别匹配 API 密钥、密码或连接字符串模式的硬编码字符串"替代"找出安全问题"。

可衡量标准的好处：

- 使跨运行结果保持一致
- 允许自动化验证（可以验证被标记的函数确实超过 50 行）
- 通过缩小范围减少误报
- 建立开发者对系统的信任

### 代码示例 — explicit-criteria.py

```python
# Vague: inconsistent results and over-flagging
vague_prompt = """
Review this code for quality issues.
Be thorough and flag anything suspicious.
"""

# Explicit: consistent, actionable results
explicit_prompt = """
Review this code and flag ONLY the following:

1. Functions exceeding 50 lines of code
2. Async operations missing try-catch error handling
3. Hardcoded strings matching API key patterns (sk-, pk-, key-)
4. Public functions missing JSDoc documentation
5. SQL queries built with string concatenation

For each finding, provide:
- File path and line number
- Rule violated (1-5)
- Severity: critical (rules 3,5) | warning (rules 1,2) | info (rule 4)
- One-line fix suggestion
"""
```

### 考试提示

每当考试问到关于生产系统的提示设计时，正确答案总是使用具体、可衡量的标准。如果某个选项说"要彻底"或"找出所有问题"，那就是错误答案。

## d4.2 少样本提示

### 概要

使用少样本示例来引导 Claude 的输出格式和推理方式。了解何时提供示例以及提供多少个。

### 少样本提示技术

- **2–4 个示例**：对于模糊情况建立格式和推理模式的最佳数量
- **格式一致性**：所有示例应遵循相同的输出结构
- **边缘案例覆盖**：至少包含一个处理边缘案例的示例
- 当任务具有模糊边界时，少样本提示最有价值

### 应避免的反模式

- 过多示例（>6 个）导致提示臃肿而无增值
- 示例间格式不一致，使模型困惑

### 深度解析

**少样本提示**提供 2–4 个示例，建立预期的输出格式、推理模式和边缘案例处理方式。当任务边界模糊时最具价值。

**少样本提示的黄金法则：**

- **2–4 个示例**——少于 2 个无法建立模式；超过 4–6 个则臃肿且无成比例收益
- **格式一致性**——所有示例必须遵循完全相同的输出结构
- **边缘案例覆盖**——至少一个示例应展示模糊或边缘情况
- **多样性**——覆盖不同类别（正面/负面/中性，简单/复杂）

**少样本提示最有价值的场景：**

- 模糊的分类任务（含讽刺意味的情感分析、混合评价）
- 非标准的自定义输出格式
- 特定领域的推理模式
- 类别边界模糊的任务

**少样本提示不必要的场景：**

- 简单、定义明确的任务（如"提取邮箱地址"）
- 具有清晰客观标准的任务
- Claude 原生支持的标准输出格式（JSON、XML）

### 代码示例 — few-shot.py

```python
few_shot_prompt = """
Classify customer reviews. Provide sentiment and reasoning.

Example 1 (clearly positive):
Input: "Absolutely love this product! Best purchase this year."
Output: {"sentiment": "positive", "confidence": "high",
         "reasoning": "Strong positive language with superlatives"}

Example 2 (clearly negative):
Input: "Terrible experience. Product broke after two days."
Output: {"sentiment": "negative", "confidence": "high",
         "reasoning": "Explicit negative evaluation + product failure"}

Example 3 (ambiguous - mixed sentiment):
Input: "Great features but battery life is disappointing."
Output: {"sentiment": "mixed", "confidence": "medium",
         "reasoning": "Positive on features, negative on battery"}

Example 4 (edge case - sarcasm):
Input: "Great, another update that breaks everything."
Output: {"sentiment": "negative", "confidence": "medium",
         "reasoning": "Sarcastic positive language masking frustration"}

Now classify this review:
Input: "{user_review}"
"""
```

### 考试提示

考试会测试你是否知道少样本示例的最佳数量（2–4 个），以及至少一个示例应覆盖边缘案例。超过 6 个示例永远是错误答案——这只会使提示臃肿而无增值。

## d4.3 用于结构化输出的工具使用

### 概要

使用 tool_use 来保证 JSON schema 合规性。理解 schema 合规性与语义正确性之间的区别。

### 通过 tool_use 实现结构化输出

- **tool_use 保证 JSON schema 合规性**——输出将匹配定义的结构
- **语义错误仍然可能发生**：结构正确但内容可能错误
- **tool_choice 选项**：`"auto"`、`"any"` 或强制指定工具以确保调用
- **Schema 设计**：必填 vs 可选字段，带 `"other"` + 详情的枚举，可空字段

### 应避免的反模式

- 假设 tool_use 消除所有错误（它仅保证结构合规）
- 未对有意外值的字段使用带 `"other"` 类别的枚举

### 深度解析

**tool_use** 是从 Claude 获取结构化输出的最可靠方式。通过定义带 JSON schema 的工具，可以保证输出匹配 schema 结构。

**关键区别：**

- **tool_use 保证结构（STRUCTURE）**——所有必填字段存在、类型正确、枚举值有效
- **tool_use 不保证语义（SEMANTICS）**——值可能错误（错误的人名、错误的日期等）

这意味着提取后仍需要验证。Schema 确保获得有效的 JSON 对象，但内部内容可能包含错误。

**tool_choice 参数：**

- **`"auto"`**——Claude 自行决定是否使用工具（默认，通用目的）
- **`"any"`**——Claude 必须使用工具，但可选择使用哪个
- **`{"type":"tool","name":"X"}`**——强制使用特定工具（用于提取，保证 schema 合规）

**Schema 设计最佳实践：**

- 始终为必填字段添加 `"required"`
- 对已知值的分类字段使用 `"enum"`
- 在枚举中包含 `"other"` 类别 + 详情字段以应对边缘情况
- 对可能缺失的字段使用 `["string", "null"]`
- 为每个属性添加 `"description"` 以提高清晰度

### 代码示例 — tool-use-extraction.py

```python
import anthropic
client = anthropic.Anthropic()

extract_tool = {
    "name": "extract_invoice",
    "description": "Extract structured data from invoices",
    "input_schema": {
        "type": "object",
        "properties": {
            "vendor_name": {"type": "string"},
            "invoice_number": {"type": "string"},
            "date": {"type": "string", "description": "ISO 8601 format"},
            "total": {"type": "number"},
            "document_type": {
                "type": "string",
                "enum": ["standard_invoice", "credit_note",
                         "proforma", "other"]
            },
            "document_type_detail": {
                "type": "string",
                "description": "Required when document_type is 'other'"
            }
        },
        "required": ["vendor_name", "invoice_number",
                     "date", "total", "document_type"]
    }
}

# Force this specific tool = guaranteed schema compliance
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    tools=[extract_tool],
    tool_choice={"type": "tool", "name": "extract_invoice"},
    messages=[{"role": "user", "content": f"Extract data from: {invoice}"}]
)
```

### 对比：反模式 vs 正确做法

| 反模式 ❌ | 正确做法 ✅ |
|-----------|------------|
| 假设 tool_use 捕获所有错误——提取数据后直接存入数据库，无验证。"来自 tool_use，肯定正确！"——结构有效但 `vendor_name` 可能错误 | 在 tool_use 之后验证语义——结构已保证，但验证内容：日期格式是否正确、总金额是否大于零、如有错误则附带错误信息重试 |

### 考试提示

tool_use 保证的是**结构（STRUCTURE）**，而非**语义（SEMANTICS）**。考试会出现声称 tool_use 消除所有错误的选项——那永远是错误的。你仍需验证提取出的值。

## d4.4 验证-重试循环与多轮审查

### 概要

实现验证-重试模式和多轮审查策略以获得可靠输出。理解重试何时有效、何时无效。

### 验证与审查模式

- **验证-重试循环**：将具体错误附加到提示中并重试以实现自我纠正
- **detected_pattern 字段**：追踪驳回模式以识别系统性问题
- **多轮审查**：逐文件本地分析 + 跨文件集成审查
- **自我审查的局限性**：同一会话会保留推理上下文，降低审查有效性
- **批处理**：阻塞任务用同步，延迟容忍负载用批处理

### 应避免的反模式

- 同会话自我审查（模型保留其推理上下文，造成盲点）
- 通用重试而不附加具体错误信息
- 汇总准确率指标掩盖了按文档类型的失败情况

### 深度解析

**验证-重试循环**和**多轮审查**是提升输出质量的生产级模式。

**验证-重试循环：**

1. 使用 tool_use 提取数据
2. 根据业务规则验证输出
3. 如果验证失败，**附加具体错误详情**并重试
4. 模型基于显式反馈进行纠正
5. 使用 `detected_pattern` 字段追踪系统性失败

**关键原则：具体的错误反馈，而非通用的。**

| 错误做法 | 正确做法 |
|---------|---------|
| "有错误，请重试" | "行项目总计（$450）与小计（$500）不匹配。税费字段包含百分比（10%）而非美元金额。" |

**多轮审查策略：**

- **第一轮（局部）**：独立审查每个文件——捕获语法、命名、缺失的错误处理
- **第二轮（跨文件）**：审查文件间的交互——捕获损坏的导入、接口不匹配

**同会话自我审查的局限性：**

当同一个会话生成和审查代码时，它会保留原始推理上下文，造成盲点。解决方案：使用**不同会话**进行生成和审查。

**批处理策略：**

| 任务类型 | 方案 |
|---------|------|
| 阻塞式 PR 审查 | 同步 |
| 夜间代码审计 | Batch API（节省 50%） |
| 实时反馈 | 同步 |
| 每周合规扫描 | Batch API（节省 50%） |

### 代码示例 — validation-retry.py

```python
def extract_with_validation(document, max_retries=3):
    messages = [{"role": "user", "content": f"Extract data from: {document}"}]

    for attempt in range(max_retries):
        response = client.messages.create(
            model="claude-sonnet-4-20250514",
            tools=[extract_tool],
            tool_choice={"type": "tool", "name": "extract_invoice"},
            messages=messages,
        )

        data = parse_tool_response(response)
        errors = validate(data)

        if not errors:
            return data  # Valid - return result

        # Critical: attach specific error details for retry
        messages.append({"role": "assistant", "content": response.content})
        messages.append({
            "role": "user",
            "content": f"Validation failed. Correct these errors:\n"
                + "\n".join(f"- {e}" for e in errors)
                + "\nPlease fix and re-extract."
        })

    raise ExtractionError(f"Failed after {max_retries} attempts")

def validate(data):
    errors = []
    if data["total"] <= 0:
        errors.append(f"Total must be positive, got {data['total']}")
    if sum(i["total"] for i in data["line_items"]) != data["subtotal"]:
        errors.append("Line item totals don't match subtotal")
    return errors
```

### 考试提示

三个必须知道的事实：
1. 重试时附带**具体**的错误详情，而非通用消息
2. 同会话自我审查是一种反模式
3. 对非紧急任务使用 Batch API（节省 50% 费用）

## 领域 4 考试提示汇总

1. **显式、可衡量的标准** > 模糊指令（始终如此）
2. **2–4 个少样本示例**是模糊任务的最佳数量
3. **tool_use = 结构合规性**，而非语义正确性
4. **同会话自我审查是反模式**——应使用不同会话
